Chapter 7 - Potato and potahto

Where do little formulas come from?

In chapter 1 we established that if math is a language, then it is for talking about counting, and counting has to do with numbers (eventually written in unary).
We also encountered rules to rewrite a math expression in different ways (in chapter 5); for example we found out that: ( a + b ) * k = k*a + k*b where a, b and k are any three numbers. The "=" sign here does not mean an equation (like it did in chapter 6) but instead that one side is THE SAME as the other.
In math these rules are called fomulas, and the idea is that I can always replace the left part of a rule with the right (and vice versa too), in any math expression: its look (AKA syntax) will change but not its meaning (AKA the semantics).
Inventing formulas is one of the main things you do when you work with math. And often formulas come from the fact that you want to count stuff, and that stuff has a structure. For instance in chapter 1 we wanted to count sequences of Is (or sheeps if you prefer) and ended up inventing whole numbers and the base 10 system (abacus and all).
Sequences of tallies have a very simple structure: they are just one I after the other. In chapter 2 we looked at Peano's defintion of numbers, and found that the main characteristic of a whole number is that there can be many Is in it, and that whole numbers come one after the other in single file, so that each one has a successor and predecessor (zero excluded of course).

So can I find a math expression that connect the number of squares (call it variable s) with the total number of matchsticks it takes to make that many squares (call it m)?
This expression will work like a kind of machine: the number of squares that I want to build goes in, and out comes the number of matches I need!

numberOfSquares --> [ machine ] --> numberOfMatchesNeeded

Let's go case by case, to get the feeling of things:

when s=2 the machine should reply 1+3+3 = 4+3 = 7,
when s=3 the machine should reply 1+3+3+3 = 7+3 = 10, ...

Note: s cannot be zero or less, because we have not defined what that means here. So s can only be a whole, positive number.
OK, good. In math notation, they would call the machine a function that defines a relation between s and m: numberOfMatchesNeeded( s ) = 1 + 3 * s or for short: matches(s) = 1 + 3 * s. The function definition reads like "the number of matches, that depends on the variable s, is calculated by 1+3*s"; and matches(s) is read as "matches of s".
Terrific. Say that you want to build 5 squares; (spoiler!) according to the table above you would need 16 matchsticks. Using our function, I can calculate how many matchsticks you would need: 1 + 3 * 5 → 1 + 15 → 16 which is correct. So our function works like a shortcut, that tells me how many matchsticks I need to build a certain number of squares, without having to draw the actual squares and count. Cool.
And the function itself is our old friend, the linear combination, in this case of one and three with the variable s. In fact, in chapter 2 we played with a linear combination of the form a*10+b and we discovered that "a linear combination can also be a way to re-order, to count numbers in a different order".

In math it is very important to try and write the same things in multiple ways, eventually ways that look different but really mean the same. You might have noticed that in this book I keep going back to that idea, and here it is again; a kind of double vison in which writing (or drawing) a thing in two or more ways actually does help in finding out more about it.

Well... this is an equation, and both sides (to the left and right of the "=" sign) are expressions with the variable n. So I can look back at what we did in chapter 5 (the one about polynomials) and chapter 6 (about equations), and I should be able to see if the two sides of the equation are THE SAME. The playground might be convincing, but trying with some values (even A LOT of values) is NOT a mathematical proof, and should not be enough to make me completely sure that my equation is true for any whole number n... Luckily, here I have already proved that it is, by calculating and showing that the two sides of the equation are equal.
Still! Testing with numbers is a good way to get a FEELING for how this formula works, what it says about the geometry of the sliced-up square.

A step back from geometry

I found a couple of ways to define the area of a square (AKA n2), using a smaller square (AKA (n-1)2) and some other pieces. But the two equations that I found can also be seen as general rules for re-writing the NUMBER n2.
In fact the two equations tell me that the square of a whole number like 7 can be calculated without using 7*7 (AKA 72), but using 6 and 62:

n2 = (n-1)2 + n + (n-1) → 7*7 = 6*6 + 7 + 6

n2 = (n-1)2 + 2*(n-1) + 1 → 7*7 = 6*6 + 2*(6) + 1

both give 49 as result. And the second formula is even cooler the first, in a way, because it never uses 7, only 6 and 1! One could look at these kind of rules as SHORTCUTS: it might save time to calculate the square of a number, via a smaller number.
We derived these two equations from a square of dots (AKA from some things that are true about the geometry of squares), but now we want to FORGET that, and focus on what they say about a generic number n, AKA the variable n. And with that in mind, I might ask: would the two equations keep being true also for negative values of n?
Probably not, and not just because I cannot draw a square that has -3 dots on its four side; the problem is the n2 part of the formulas, and a negative number to the second power (AKA a number squared) becomes positive, messing up our equations. It's worth trying with -3 for both, just to see what happens. With n=-3 the first equation becomes:
n2 = (n-1)2 + n + (n-1) →
(-3)2 = ((-3)-1)2 + (-3) + ((-3)-1) →
9 = 9 -3 -4 → 9 = 2 → false
and since I wanted to check if my equation worked for all numbers (positive and negative), one bad example is enough to be sure that IT DOES NOT. It still works for all positive, whole numbers, but I have to exclude the whole numbers that are negative.
Now I can try the second equation with n=-3:
n2 = (n-1)2 + 2*(n-1) + 1 →
(-3)2 = ((-3)-1)2 + 2*((-3)-1) + 1 →
9 = 9 + 2*(-4) + 1 → 9 = 9 -8 + 1 → 9 = 2 → false
which means also this equation works only for positive numbers, zero included.

So when I create my own rules, like
"something can be re-written as, or replaced by something else"
I have to be extra careful that they are true; and if a rule is only true in some cases, I need to be very explicit about which cases they are. Here I can be sure that n2< can be replaced by (n-1)2+n+(n-1), and also by (n-1)2+2*(n-1)+1, in any other math expressions, BUT only when n is a whole number equal or larger than zero.

Moral of the story so far: the examples of slicing squares show that it is perfectly OK to make up some weird relationship between certain numbers, and ask "is this true in general?".

Problems within problems

Formulas can also be created when a variable is substituted by an entire expression.
Wait... when would that be needed? Putting a math expression in the place of a variable?
Well, it could make sense if a problem is based on another problem. For example, consider the problem of finding out how many matchsticks I need to build a certain number of squares (that we solved in the beginning of this chapter). I a way we don't only solved that problem, but we also created a sequence of numbers: the numbers generated by the matchsticks squares. The first of those matchsticks squares used 4 matches, the second used 7, then 10, and 13, and so on. So the sequence generated is: 4, 7, 10, 13, ....
Say that I have another problem, and that I need to use that sequence, to solve my new problem. Here it is: I want to build a lot of dots squares, with sides based on the sequence above. And I am interested in finding a function that would give me the area of one of these dots squares, given its index in the sequence. 

4 matches   7 matches     10 matches     13 matches     ...
                                      ...
  ↓

Area = 16

Let me be more concrete. The smallest of these new squares should have side equal to 4, and an area of 4*4 = 16; the second's side should be 7, and the area 7*7 = 49. Then a side of 10, and an area of 10*10 = 100, and so on.
In general, to know the area of one such square, I need to take n, calculate the nth number in the matchsticks sequence, and then calculate its square.
The function that gives the matchsticks sequence numbers, given s, is matches(s) = 1 + 3 * s only that here s should really be called n. So I can substitute s with n: matches(n) = 1 + 3 * n
And the new function, that I want to call squaresOfSquares, should of the form something2. But I need to substitute the variable something by matches(n). The result of that substitution is: something2 → matches(n)2 → (1+3*n)2
squaresOfSquares(n) = (1+3*n)2 So now I have my function squaresOfSquares(n), that takes a number n and gives me the area of the square of dots, that has the sides of lenght matches(n). This new function is made up of two other functions composed together.
Let's to get an idea of how this function.

So what the function squaresOfSquares(n) means is "what is the area of that square?", or if you like: "how many dots are in a square with a side based on the sequence of matchsticks squares numbers?"
In math, writing someFunction(5) (for example he had squaresOfSquares(n)) is "CALLING the function someFunction on the value 5". Now assuming that the function someFunction is defined something like: someFunction(k) = some complicated math expression involving the varabile k the result of CALLING someFunction(5) is to substitute the variable k in the defintion with the value of the CALL, here 5. So I will have: someFunction(5) = some complicated math expression involving s5 On the hand, a function call is going to become a number, when the function is calculated; in this example someFunction(5) might turn out to be the number 123, and that is fine. But it also mean that I can treat the call to a function as a un-calculated number, another of the lazy numbers that we discussed in this book. If you are with me so far, then you should agree that it is OK to write something like: someFunction(5)+5 or: 12*someFunction(5) and: 99 + someFunction(a) + someOtherFunction(b) or even: someFunction( someOtherFunction(a) + b ) * 99 where a and b could be numbers or variables.

I find it also very interesting to see that even a function can be written in multiple ways, and still mean the same thing. For example we just used the fact that: matches(s) = 1+3*s is THE SAME as matches(n) = 1+3*n I just have to be sure that the new variable n is not present in the original function, otherwise I would mess things up.
We even composed two functions togehter: squaresOfSquares(n) = (1+3*n)2, which can be seen as a couple of machines like this:

s --> [ 1+3*s ] --> numberOfMatchesNeeded

a --> [ a2 ] --> squareOfN

connected together. And the result is a new, LARGER machine, that uses both to calculate my new function squaresOfSquares(n):

n --> [ s --> [ 1+3*s ] --> a --> [ a2 ] --> ] --> squaresOfSquares

This machine is equivalent (AKA it looks different but it does the same for any possibly input) to this other machine, that I got by putting two math expressions together:

n --> [ (1+3*n)2 ] --> squaresOfSquares

So also machines (AKA functions) can be transformed to look different but keeping their meaning. Hence, there is probably some kind of normal form for functions that I could define, and using some meaning-preserving transformations, rewrite functions until the are in normal form.
For instance these two functions:
i --> [ i ] --> theResult

j --> [ j+j+j3 ] --> anotherResult
do the same thing, but the first looks SIMPLER, so it is possibly closer to a normal form, than the second. Rewriting functions to look simpler is very important not only in math but also in computer programming.

A square peg in a round hole

We have made up a few math rules in this chapter. For example: (n+m)2 = n2 + 2*n*m + m2

n2-m2 = (n+m)*(n-m)
But when should I use a math rule (AKA formula)? Well, is not difficult to decide: for example when I have two variables and I need to calculate their square, I can use the first of the two formulas. If I have (apple+banana)2, I can just subtitute n with apple and m with banana and write: (3*apple+4*banana)2 = (3*apple)2 + 2*(3*apple)*(4*banana) + (4*banana)2 automatically, without even thinking... typographically if you like. The of course I will have to do a few calculation, to make my result LOOK a bit better: 9 * apple2 + 2*3*4 * apple*banana + 16 * banana2 →
9*apple2 + 24*apple*banana + 16*banana2
In cases like this I'm using a rule left-to-right, or forward, and all I have to do is to substitute my expressions to the original variables of the formula (and perhaps calculate a bit, to clean up). This is not very different than what we discussed above about calling a function.
Fine. But what the other way around? What about having a long expression and looking to FIT IT in a certain template, so I can use a rule and rewrite my original expression?
Confused? Sorry. Let me find an example.
Say that you are trying to solved a problem, and after writing your problem down as a math expression, you are left with expression: 9*x2 + 12*x + 4. And for some reason you would very much like to rewrite that as the square of something, AKA (something)2, where your something is possibly a math expression that contains x.
From the rule above you know that something like this: n2 + 2*n*m + m2 can in fact be rewritten like: (n+m)2. For sure.
So... how do you fit your "square peg" 9*x2 + 12*x + 4 into the "round hole" n2 + 2*n*m + m2 ? Do these two expression even match? And assuming things fit, and I can rewrite my initial expression a as: (n+m)2 what would n be here and what would m be?
To use a formula backward I need change the look of an expression until it looks like another expression; and all the transfromations that I will use in the process, should not change what the first expression meant. (Sounds familiar? Like when we discussed normal forms in chapter 1 or chapter 6...? Just saying...).
So how can I make 9*x2 + 12*x + 4 look like n2 + 2*n*m + m2 ? Well, luckily they are not VERY different to start with!
Unless I can "see" how to rewrite the first expression to have the same shape, form as the second, I might take an experimental approach: try and MUTATE the first expression, see in how many ways I can re-write it without changing its meaning, then find the look that seems more promising to have a close match with the second expression. Finally, I can try to fit the MUTATED expression in the template (AKA the second expression, AKA the "round hole"). Let's play this mutation game:

Another misfit

Now we have a technique to FIT an expression into a template, and that allow us to use the math rules that we have discovered. Cool.
Let's test this new idea of fitting an expression to a template, with another square: ( egg + salt + bacon )2 Or if you like it simpler: (a+b+c)2.
But what should I do here? After all our rule works when there are two variables: (n+m)2. How can I put 3 feet in 2 shoes? Because that's what ( egg + salt + bacon )2 looks like, with respect to the template (n+m)2.
What are n and m here? When I have three things, egg, salt and bacon!
Perhaps I can CHEAT a bit and group two things, and pretend I only had two to start with... But would it work? Let's see: ( egg + salt + bacon )2 → ( (egg+salt) + bacon )2 OK, so now it is of the form: ( someThing + someOtherThing )2 which is promising. I can now use the rule and see where it gets me: ( (egg+salt) + bacon )2 → n2 + 2*n*m + m2
and after substituting it becomse: (egg+salt)2 + 2*(egg+salt)*bacon + bacon2 OK, it worked! Only... now I still have a square to calculate, and some other minor stuff... (egg+salt)2 + 2*egg*bacon + 2*salt*bacon + bacon2 The first part of the expression is (egg+salt)2 which fits nicely in the same template, so I can use the same rule again: ( egg + salt )2 → n2 + 2*n*m + m2 →

egg2 + 2*egg*salt + salt2 Putting it all together: (egg2 + 2*egg*salt + salt2) + 2*egg*bacon + 2*salt*bacon + bacon2 and I just have to clean it up: egg2 + salt2 + bacon2 + 2*egg*salt + 2*egg*bacon + 2*salt*bacon Done.
